How to get started with the CCF
 
 
Thank you for trying out the CollabNet Connector Framework (CCF). The CCF SDK is an openAdaptor based SDK that allows rapid integrations and migrations dealing with the artifact data shared between different tools in the ALM cycle in combination with the collaborative platforms from CollabNet.
This page will point you to the information you need in order to learn more about the CCF. You can read these documents step by step or just jump directly to the documents you are mostly interested in.
 
 
General Information about the CCF
 
 
Most of the requirements and descriptions, what the CCF should be can be found in our Wiki. The detailed goals and the purpose of the CCF can be found there as well as its anticipated customers, stakeholders and clients.
If you are interested in using the CCF but not yet sure whether it was designed for your specific profile, please look in the user categories of the CCF to find out.
 
 
The environment, CCF is typically used in
 
 
If you read the general information about the CCF you might still be wondering whether the environment your integration will take place in is supported by the CCF. The Wiki again can help to see the constraints of the environment, CCF was designed for.

Underlying technologies of the CCF
 
 
The CollabNet Connector Framework is heavily based on openAdaptor and the (self explaining) component wiring configuration language of the Spring Framework. As long as you just want to use the CCF for some of the many preconfigured and prepackaged scenarios which come with the standard distribution of the CCF, you do not have to be familiar with openAdaptor at all.
If you are interested in implementing your own connectors to third party tools or want to use some of the many predefined openAdaptor components (file readers, file writers, access to various middlewares and transaction systems, JMS, JMX, data bases, web services, JNDI, FTP, SMTP, Servlets, XSLT processors, Schema validators, object streamers, message filters based on velocity and java script, ...) within your integrations without having to write a single line of code, you should take a look in the white paper of openAdaptor (eight-pager) and this excellent article.

 

Predefined scenarios supported by the CCF

 

The fastest way to get started with the CCF is probably by trying the preconfigured example integration scenarios. All you have to do is just to change some configuration properties (like URL of your SFEE site, trackers to be synchronized and user credentials) and watch CCF do its synchronization job. Once you managed to run the examples you can start looking into the example wiring files (fully documented) to find out how the scenario was actually accomplished and try to change some further aspects. We also encourage you to try to combine the features of multiple examples (just copy paste the features you need).

If you want to achieve an integration scenario that goes far beyond the predefined scenarios, it might be the time to read the section above (about the underlying technologies) and learn more about openAdaptor. After this, you will have enough knowledge how to integrate further components into your wiring files to get the exact integration you want. It may also help to read the section below dealing with the specific terms used within the CCF.

 

The Generic Artifact XML Format

 

After trying out the basic example you will find out that in order to synchronize your own trackers, you actually have to specify (e. g. by using an XSLT file) how the artifact fields of your source artifact have to be transformed to fit the format of the target artifacts (for explanation of the term artifact, please read the following section). The CCF already abstracts from the vendor specific artifact and repository format so you do not have to concentrate on implementation specific details.

All you have to understand (to some point) is one XML based artifact format that is used by all CCF compliant components. This format is called Generic Artifact XML Format and is defined based on an fully documented XML schema you can find here.

Don't panic: A simple integration will only use a minority of the features defined within this specification. The predefined examples already come along with lots of transformation files doing things like value substitutions, field deletion, field renaming and adding of constant values, so you can just use these XSLT files and adjust them to fit your needs.

Once you basically understood these transformation files and the generic artifact format (you can also use the predefined Generic Artifact Export scenario to get an idea how the artifacts of your tracker look in the generic artifact xml format), you can start specifying your own artifact transformations the CCF will execute for you.

 
Technical Terms and Abbreviations used within the CCF
 
 
The CCF Wiki also contains the technical terms and abbreviations that are typically used within the technical discussions about the CCF. These terms are very important to know if you decided to write your own CCF compliant components (what a CCF compliant component is, is also explained there), like to participate in the core development of our framework or just like to understand the concepts of the CCF in depth. Also if you like to understand the ongoing technical discussions on our developer mailing list it is probably a good idea to know what polling cycles, identity mapping tables, sychronization info records or entity services actually are.
 
Developer documentation for the CCF
 
 
If you read through all the wiring files of the predefined example scenarios and had a look into our glossary of terms, you should be able to understand the source code of the CCF as well. The source code can be found within our SVN repository which can be browsed online as well as the generated javadocs. If you have any questions regarding the code, discovered a bug, like to contribute a new reader or writer component or just want to get in touch with us, just join our developer mailing list.
