CollabNet Connector Framework

 


"What is the CCF and why do I need it?"

When we are asked, what the CCF is, we usually give two short answers.

Short formal answer: "The CCF SDK is an openAdaptor based SDK that allows rapid integrations and migrations dealing with the artifact data shared between different tools in the ALM cycle in combination with the collaborative platforms from CollabNet."

Short informal answer: "When talking about the CCF, think about a freeway. This freeway is not for transporting cars but for transporting ALM related data. For every new tool that likes to use the freeway, you just have to provide a ramp and your tool data can travel anywhere (into any other tool connected with a ramp to the freeway)."

"So what's so nice about the CCF?"

In terms of our freeway metaphor it is "not building the freeway twice" or even a little bit more general "not inventing the wheel over and over again". If you think back how integrations between tools are usually done, you probably agree that most of them are actually point-to-point integrations. If you build a point-to-point integration between tool A and tool B, you typically have to deal with three steps:

  1. Define, design, build and test the components used to extract and write data into tool A (a ramp in our freeway methaphor)
  2. Define, design, build and test the components used to extract and write data into tool B (a ramp in our freeway methaphor)
  3. Define design, build and test the components used to synchronize the data between the tools, care about identity mapping, data mapping, transactionality, error recovery, logging, locking, usability, configuration, user interaction, security, load balancing, conflict resolution, transport protocols, meta data changes, documentation, throttling, streaming, installation, packaging, testing, monitoring, ... (an ordinary street in our freeway metaphor)

According to our experience, it is a valid approach to conclude from the length of the bullet points to the amount of time spent for each of these steps.

One point-to-point integration is usually not the only integration within a bigger company. In a complex application lifecycle scenario, the data of one tool has to be transported to numerous other tools, so there is the need for many tool integrations. If we now also like to connect tool A with tool C we would have to repeat step one to step three again if we used the point-to-point integration approach again. Especially step 2 and step 3 seem to be duplicated work but unfortunately this is often the case of point-to-point integrations built by different contractors or development teams.

By using the CCF for our integration, we only have to build another ramp for tool C and we are done. Every functionality described in step 3 is core functionality of the CCF and can be considered as our "data freeway". This functionality (mapping, logging, routing, ...) can be used by every "car" travelling over our our freeway no matter from which "ramp" it comes from and which "ramp" it likes to use to exit the freeway.

If you read the last sentence, you probably noticed that by providing a ramp for tool C, we now also have the possibility to synchronize tool B and tool C with each other without writing any further code. Just think about the possibilities you get if you do not just have three but twenty ramps!

 

"So how does it work?"

So if by just providing a ramp for a new tool, the data from this tool can be synchronized with any other tool, does this mean that the data format always have to look the same? When you think about a real freeway, not all cars look exactly the same and they can still use the same ramps. However, there are certain restrictions for every car when it comes to its dimensions, license plates, speed, etc.

Similar constraints apply to the artifacts (data packets) travelling the CCF freeway. They all have to follow a well defined XML format. This format does not constrain the number, values and characteristics of the tool data but it defines the way, this data is represented. Using this representation, every ramp is able to receive artifact data from any other ramp. The representation is also important to correctly "route" the artifact data to the correct exit ramp (think about traffic signs showing directions on the freeway). The representation format is called "Generic Artifact (XML) Format".

If you like to use the CCF, there is a high chance that there is already a ramp for the tools you like to synchronize. The core CCF provides ramps for SourceForge Enterprise Edition, CollabNet Enterprise Edition, flat delimited text files and data base tables, or to express it in a more formal way (cited from our requirements Wiki):

"The CCF SDK contains components used to extract artifacts from SourceForge Enterprise Editions (SFEE) and CollabNet Enterprise Editions (CEE) repositories and transform them to a canonical XML based format. [...] The CCF SDK also contains components that can understand artifact data represented in the Generic Artifact XML format and write them back into SFEE and CEE repositories."

If your favourite tool is still not supported, all you have to do is to write a new ramp that understands the "Generic Artifact (XML) Format." Since the CCF is based on a very powerful "ramp construction kit" called openadaptor, building new ramps should be really straight forward.

Once your "cars" (artifact data) are on the freeway you just have to specify the transformations to be applied on them and where they should exit the freeway again. The freeway itself consists of multiple core CCF components that were constructed following the best practices of numerous point-to-point integrations done at CollabNet and other companies. You do not have to use all core components of the CCF for your particular integration scenario, but just the ones you need. Fr instance, as long as you just mirror data, you do not have to use any data transformation component. Consider the CCF as a toolbox of ramps and freeway building blocks you can combine however you want without having to write a single line of code, or to express it in a more formal way (cited from our requirements Wiki):

"The CCF SDK contains components that solely work on the generic artifact XML representation and provide system independent support functions to facilitate the synchronization process (e. g. identity mapping, artifact data transformation and CSV file to generic artifact convertors). [...] All components can be used totally independent from each other and only communicate by the standard mechanisms of openAdaptor (i. e. by modifying the artifact data traveling through the openAdaptor message pipeline). They can be used without adaption with any other standard openAdaptor component as as with any third party component following the generic artifact XML format specification for its input and output. "

The process of wiring the different components together (think about putting the correct traffic signs on the freeway) is not complicated at all. We already provide lots of standard integration scenarios within the CCF download. You may just change some property values to see CCF in action within your own system landscape, so to cite from our Wiki again:

"The CCF SDK contains documentation and examples how to wire the independent components together in order to achieve a certain integration scenario (e. g. SFEE to SFEE tracker synchronization or CSV file to CEE tracker synchronization)."
